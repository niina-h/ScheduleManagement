## 3. コーディングルール

詳細は `.clinerules` を参照。以下はここで完結する最重要ルール。

### 3-1. 変更スコープ
- **指示された箇所のみ** 修正する。無関係なリファクタリングは行わない。
- 変更前後の差分を必ず説明してから適用する。

### 3-2. コメント・型ヒント
- すべての公開関数・クラスに日本語 docstring を付与する。
- 関数引数と戻り値には型ヒント（`typing` モジュール）を必ず付ける。

### 3-3. エラー処理
- 例外は握り潰さず、ログ出力またはユーザー向けメッセージを必ず残す。
- `except Exception` の使用は最終手段とし、具体的な例外型を優先する。

## 4. セキュリティ要件

### 4-1. ファイル操作
- ユーザー指定のファイルパスは **`pathlib.Path` で正規化** し、プロジェクトルート外へのアクセスを禁止する。
- CSVファイルは `encoding='utf-8-sig'` で読み込み、文字化け・BOM問題を防ぐ。
- ファイル書き込みは **一時ファイル → アトミックなリネーム** の手順で行い、書き込み中のクラッシュによるデータ破損を防ぐ。

```python
# 推奨パターン
import tempfile, pathlib, os

def safe_write(path: pathlib.Path, content: str) -> None:
    tmp = path.with_suffix(".tmp")
    tmp.write_text(content, encoding="utf-8-sig")
    os.replace(tmp, path)  # アトミック
```

### 4-2. 入力バリデーション
- 日付文字列は `datetime.strptime` で厳格にパースし、生の文字列比較は行わない。
- CSVの各フィールドは最大長・許容文字セットをバリデートしてから処理する。
- GUI入力値は `str.strip()` → 空文字チェック → 型変換の順で必ず検証する。

### 4-3. ログ・機密情報
- ログにファイルフルパス・個人情報を **そのまま出力しない**（必要なら末尾N文字のみ）。
- `print()` によるデバッグ出力は本番コードに残さず、`logging` モジュールに統一する。

## 5. トークン節約ルール（Claude Code 向け）

Claude Codeがコンテキストを効率よく使えるよう、以下を守ること。

### 5-1. 読み込み対象の限定
- **変更に直接関係するファイルのみ** を読み込む。プロジェクト全体を一括スキャンしない。
- 1回の指示で読み込むファイルは **最大5ファイル** を目安とする。

### 5-2. 差分ベースの作業
- 修正は **`str_replace` 形式（before/after）** で提示する。ファイル全体の再出力は避ける。
- 大きな変更は複数の小さなステップに分割して段階的に適用する。

### 5-3. 不要な出力を省く
- コード説明は **変更点の要約のみ**。変更のない部分の解説は省略する。
- テスト結果は合否と失敗原因のみ報告し、ログ全文は貼り付けない。

### 5-4. コンテキスト汚染の防止
- 同一会話セッションで複数の無関係タスクを混在させない。
- タスクが切り替わる場合は新しいセッションを開始するよう促す。

## 6. 作業フロー

```
1. 要件確認   → 曖昧な点を日本語で質問してから着手
2. 影響範囲   → 変更対象ファイルと影響するモジュールを列挙
3. 実装       → 差分ベースで最小限の変更
4. セルフレビュー → セキュリティ・型・docstring を自己チェック
5. 報告       → 変更概要・注意点・次のステップを簡潔に説明
```

---

## 7. 禁止事項

| # | 禁止 | 理由 |
|---|------|------|
| 1 | `eval()` / `exec()` の使用 | 任意コード実行リスク |
| 2 | ハードコードされたファイルパス | 環境依存・移植性の欠如 |
| 3 | 指示なしの依存パッケージ追加 | 意図しない環境変更 |
| 4 | テストなしの破壊的変更 | 既存機能の無言の破壊 |
| 5 | ファイル全体の無断書き換え | トークン浪費・レビュー困難 |

## 8. 開発チーム編成（マルチエージェント構成）

Claude Code のサブエージェント機能を活用し、役割を分担して並列開発を行う。  
各エージェントは **自分の担当スコープ外のファイルを書き換えてはならない**。

### 8-1. チーム構成図

```
┌─────────────────────────────────────────────┐
│              🧭 PM エージェント               │
│  タスク分解・優先順位・進捗管理・統合判断     │
└───────┬──────────┬──────────┬───────────────┘
        │          │          │
        ▼          ▼          ▼
┌──────────┐ ┌──────────┐ ┌──────────────────┐
│ Architect│ │  Dev     │ │  QA              │
│ エージェント│ │ エージェント│ │ エージェント       │
│ 設計・構造│ │ 実装・修正│ │ テスト・レビュー  │
└──────────┘ └──────────┘ └──────────────────┘
        │          │          │
        └──────────┴──────────┘
                   │
                   ▼
        ┌─────────────────┐
        │ Security エージェント│
        │ セキュリティ監査  │
        └─────────────────┘
```

### 8-2. 各エージェントの役割と担当スコープ

#### 🧭 PM エージェント（プロジェクトマネージャー）
**起動トリガー**：新機能追加・大規模変更・バグ調査の開始時

| 項目 | 内容 |
|------|------|
| 担当ファイル | `CLAUDE.md` / `.clinerules` / `docs/` |
| 主な責務 | タスクを Issue 単位に分解し、各エージェントに割り当て |
| 出力形式 | タスクリスト（番号付き）＋担当エージェント名＋完了基準 |
| 制約 | ソースコードへの直接変更は行わない |

**指示テンプレート**：
```
あなたは PM エージェントです。
以下の要件を分析し、Architect / Dev / QA / Security の各エージェントへの
タスク割り当てリストを日本語で作成してください。

要件：{USER_REQUEST}
```

---

#### 🏗️ Architect エージェント（設計担当）
**起動トリガー**：新モジュール追加・既存モジュールの大幅な再設計

| 項目 | 内容 |
|------|------|
| 担当ファイル | `src/models/` / `src/services/` の設計部分 |
| 主な責務 | クラス設計・インターフェース定義・依存関係の整理 |
| 出力形式 | クラス図（テキスト）＋インターフェース定義コード |
| 制約 | 実装コードは書かず、設計仕様書と型定義のみ作成 |

**指示テンプレート**：
```
あなたは Architect エージェントです。
以下の仕様を満たすクラス設計を提案してください。
既存の models/ と services/ との依存関係も考慮すること。

仕様：{SPEC}
変更対象：{TARGET_MODULE}
```

---

#### 💻 Dev エージェント（実装担当）
**起動トリガー**：Architect の設計承認後、または単体の軽微な修正

| 項目 | 内容 |
|------|------|
| 担当ファイル | `src/` 配下の実装ファイル全般 / `main.py` |
| 主な責務 | 機能実装・バグ修正・リファクタリング |
| 出力形式 | 差分（str_replace 形式）＋変更概要3行以内 |
| 制約 | テストコードを同時に作成する。設計変更は Architect に差し戻す |

**指示テンプレート**：
```
あなたは Dev エージェントです。
以下の設計仕様に従い、差分ベースで実装してください。
型ヒントと日本語 docstring を必ず付与すること。

設計仕様：{DESIGN_SPEC}
対象ファイル：{FILE_PATH}
```

---

#### 🧪 QA エージェント（品質保証担当）
**起動トリガー**：Dev エージェントの実装完了後

| 項目 | 内容 |
|------|------|
| 担当ファイル | `tests/` 配下のテストファイル |
| 主な責務 | ユニットテスト作成・レビュー・カバレッジ確認 |
| 出力形式 | テストコード＋レビューコメント（OK / NG / 要修正） |
| 制約 | 本番コードは変更しない。指摘は Dev に差し戻す |

**指示テンプレート**：
```
あなたは QA エージェントです。
以下の実装コードをレビューし、ユニットテストを作成してください。
pytest 形式、日本語コメント付きで出力すること。

レビュー対象：{IMPLEMENTED_CODE}
テスト観点：正常系 / 異常系 / 境界値
```

---

#### 🔒 Security エージェント（セキュリティ監査担当）
**起動トリガー**：ファイル操作・入力処理・外部連携を含む変更時

| 項目 | 内容 |
|------|------|
| 担当ファイル | 読み取り専用（監査のみ） |
| 主な責務 | セクション4のルール準拠確認・脆弱性チェック |
| 出力形式 | 監査レポート（問題なし / 要修正 + 修正案） |
| 制約 | コードの変更は行わず、Dev に修正指示を出す |

**指示テンプレート**：
```
あなたは Security エージェントです。
以下のコードを CLAUDE.md セクション4のセキュリティ要件に照らして監査し、
問題点と修正案を日本語で報告してください。

監査対象：{CODE_OR_DIFF}
```

### 8-3. エージェント間の連携プロトコル

```
新機能追加の標準フロー：

[PM] 要件分解・タスク割り当て
  ↓
[Architect] 設計・インターフェース定義
  ↓  ← PM が設計レビュー・承認
[Dev] 実装（差分ベース）
  ↓
[QA] テスト作成・コードレビュー  ←→  [Dev] 指摘修正（ループ）
  ↓
[Security] セキュリティ監査       ←→  [Dev] 脆弱性修正（ループ）
  ↓
[PM] 完了確認・ドキュメント更新
```

### 8-4. エージェント起動時の共通ルール

1. **自己紹介を省く** — 役割名は1行目に書くだけでよい（トークン節約）
2. **スコープ外は差し戻す** — 担当外の変更依頼は適切なエージェントに転送する旨を伝える
3. **完了報告は3行以内** — 何をした / 懸念点 / 次のステップ
4. **ファイル読み込みは最大5件** — セクション5のトークン節約ルールを厳守
5. **並列作業時は競合を避ける** — 同一ファイルを複数エージェントが同時編集しない

### 8-5. 簡易タスク（エージェント不要）の判断基準

以下はエージェント編成なしで直接対応してよい：

| 規模 | 基準 | 対応 |
|------|------|------|
| 小 | 1ファイル・10行未満の修正 | 直接 Dev 作業 |
| 中 | 複数ファイル・新関数の追加 | Dev + QA のみ |
| 大 | 新モジュール・設計変更を伴う | フルチーム編成 |
