## 2. デプロイ方法の選び方

```
通常のプログラム修正（バグ修正、画面変更など）
  → deploy_code.bat（プログラムのみ）

予定・実績データを移行、マスタは本番を維持したい場合
  → deploy_transaction.bat（プログラム＋トランザクション移行）

作業マスタ・ユーザー・予定データごと入れ替えたい場合
  → deploy_full.bat（プログラム＋DB上書き）
```

| バッチファイル | 場所 | DB | 用途 |
|---------------|------|-----|------|
| `deploy_code.bat` | `bat\` | 触らない | 日常的なプログラム更新 |
| `deploy_transaction.bat` | `bat\` | トランザクションのみ移行 | 予定・実績を開発から移行、マスタは本番維持 |
| `deploy_full.bat` | `bat\` | 全上書き（バックアップ自動） | データ初期化・全面入替 |

## 3. プログラムのみ更新

### 手順

```
① bat\deploy_code.bat をダブルクリック
② 接続確認が表示される → Enterキーで続行
③ 「プログラム更新完了」と表示されたら完了
④ 本番サーバーを再起動する（→ 第6章参照）
```

### 画面の流れ

```

  ※ DBは上書きしません
============================================

開発元 : C:\DEV(ClaudCode)\ScheduleManagement
本番先 : \\192.168.70.141\C$\App\ScheduleManagement

[確認] 本番サーバーへの接続を確認中...
  → 接続OK

※ サーバーの停止・再起動は手動で行ってください

続行するには何かキーを押してください . . .    ← Enterを押す

[1/3] プログラムファイルを同期中...
  → プログラム同期完了
[2/3] フォルダ構成を確認中...
  → OK
[3/3] 完了確認...

============================================
  プログラム更新完了
  ※ DBはそのまま（変更なし）
  ※ 本番サーバーの再起動を行ってください
============================================
```

### コピーされるもの

| 対象 | 内容 |
|------|------|
| `web_app\` | テンプレート・ルート・モデル・静的ファイル等すべて |
| `data\` | 初期データ（users.json等） |
| `deploy\` | デプロイ関連スクリプト |
| `bat\` | バッチファイル |
| `run_web.py` | 開発用起動スクリプト |
| `run_production.py` | 本番用起動スクリプト |
| `requirements_web.txt` | 依存パッケージ一覧 |

### コピーされないもの

| 対象 | 理由 |
|------|------|
| `db\` | 本番データを保護するため |
| `__pycache__\` | キャッシュは不要 |

## 4. トランザクション移行デプロイ

> ⚠️ **本番の予定・実績データが開発データで置き換えられます。** 実行前に本番サーバーを停止してください。
> マスタデータ（ユーザー・作業マスタ等）は本番のまま維持されます。

### 手順

```
① 本番サーバーを停止する（→ 第6章参照）
② bat\deploy_transaction.bat をダブルクリック
③ 移行内容を確認 → Enterキーで続行
④ 「デプロイ完了」と表示されたら完了
⑤ 本番サーバーを起動する（→ 第6章参照）
```

### データの分類

| 区分 | テーブル | 方針 |
|------|----------|------|
| **マスタ（維持）** | users, dept_master, task_master, task_category, task_subcategory, mail_settings | 本番データをそのまま保持 |
| **トランザクション（移行）** | weekly_schedule, daily_result, daily_comment, carryover, weekly_leave, operation_log | 開発データで上書き |

### 画面の流れ

```
============================================
  予定管理システム - プログラム＋トランザクション移行
  ★ マスタデータは本番を維持 ★
============================================

★★★ 移行内容 ★★★
  [維持] ユーザー・作業マスタ・区分・部署・メール設定（本番データ）
  [移行] 週間予定・日次実績・コメント・繰越・休暇・操作ログ（開発データ）
  [更新] プログラムファイル一式

[1/3] 本番DBをバックアップ中...
  → web_app.db.bak_20260325_143012 に保存
[2/3] プログラムファイルを同期中...
  → プログラム同期完了
[3/3] トランザクションデータを移行中...
  読取: weekly_schedule → 531 件
  読取: daily_result → 50 件
  ...
  → トランザクション移行完了

============================================
  デプロイ完了
============================================
```

### いつ使うか

- 開発環境でテスト入力した予定・実績を本番に反映したい
- 本番のユーザーマスタ・作業マスタは本番側で管理している（開発と異なる場合がある）
- DB全体を上書きすると本番のマスタ設定が消えてしまう場合

## 5. フルデプロイ

> ⚠️ **本番のDBが上書きされます。** 実行前に本番サーバーを停止してください。

### 手順

```
① 本番サーバーを停止する（→ 第6章参照）
② bat\deploy_full.bat をダブルクリック
③ 警告を確認 → Enterキーで続行
④ 「フルデプロイ完了」と表示されたら完了
⑤ 本番サーバーを起動する（→ 第6章参照）
```

### 画面の流れ

```
============================================
  予定管理システム - フルデプロイ
  ★ プログラム＋DB すべて上書き ★
============================================

開発元 : C:\DEV(ClaudCode)\ScheduleManagement
本番先 : \\192.168.70.141\C$\App\ScheduleManagement

[確認] 本番サーバーへの接続を確認中...
  → 接続OK

★★★ 警告 ★★★
  本番DBを開発DBで上書きします。
  本番の既存データはバックアップ後に上書きされます。
  本番サーバーを先に停止してから実行してください。

続行するには何かキーを押してください . . .    ← Enterを押す

[1/4] 本番DBをバックアップ中...
  → web_app.db.bak_20260325_143012 に保存
  → schedule.db.bak_20260325_143012 に保存
[2/4] プログラムファイルを同期中...
  → プログラム同期完了
[3/4] DBを上書きコピー中...
  → web_app.db をコピーしました
  → schedule.db をコピーしました
[4/4] 完了確認...

============================================
  フルデプロイ完了（プログラム＋DB上書き）
  ※ 本番サーバーの再起動を行ってください
============================================
```

### DBバックアップについて

- バックアップは本番サーバーの `db\` フォルダ内に自動作成されます
- ファイル名: `web_app.db.bak_日時` （例: `web_app.db.bak_20260325_143012`）
- 復元が必要な場合は、バックアップファイルを `web_app.db` にリネームしてください

## 6. 本番サーバーの起動・停止・確認

### 5-1. リモートデスクトップで接続

```
接続先: 192.168.70.141
```

> 以降の操作はすべて **本番サーバー上** で行います。

### 5-2. サーバーを停止する

コマンドプロンプトまたはPowerShellで以下を実行：

```cmd
taskkill /F /IM pythonw.exe
```

> `pythonw.exe` が見つからない場合は、既に停止しています。

### 5-3. サーバーを起動する

コマンドプロンプトで以下を実行：

```cmd
cd C:\App\ScheduleManagement
start /b pythonw run_production.py
```

> `pythonw` を使うとコンソールウィンドウなしでバックグラウンド起動します。

### 5-4. 動作確認

ブラウザで以下のURLにアクセスして、ログイン画面が表示されればOKです：

```
http://192.168.70.141:5000/
```

> 本番サーバー自身からは `http://localhost:5000/` でもアクセスできます。

## 7. 初回セットアップ

新しいPCに本番環境を構築する場合のみ実施します。

### 手順

```
① 本番サーバーに Python 3.10以上 をインストール
② C:\App\ScheduleManagement フォルダを作成
③ deploy_full.bat を実行（プログラム＋DBすべてコピー）
④ 本番サーバーにリモートデスクトップで接続
⑤ コマンドプロンプトで以下を実行：

   cd C:\App\ScheduleManagement
   pip install -r requirements_web.txt

⑥ サーバーを起動：

   start /b pythonw run_production.py

⑦ ブラウザで http://192.168.70.141:5000/ にアクセスして確認
```

### （任意）Windows起動時に自動実行する場合

タスクスケジューラに登録：

| 設定項目 | 値 |
|----------|-----|
| タスク名 | ScheduleServer |
| トリガー | スタートアップ時 |
| 操作 | プログラムの開始 |
| プログラム | `pythonw` |
| 引数 | `run_production.py` |
| 開始 | `C:\App\ScheduleManagement` |

## 8. トラブルシューティング

### Q. 「192.168.70.141 に接続できません」と表示される

- 本番サーバーの電源が入っているか確認
- 開発PCと本番サーバーが同じネットワークにいるか確認
- ファイアウォールでICMP（ping）がブロックされていないか確認

### Q. 「アクセスできません」と表示される

- エクスプローラーで `\\192.168.70.141\C$` に手動アクセスできるか確認
- 管理者権限のあるアカウントでログインしているか確認
- 資格情報を求められた場合は、本番サーバーの管理者アカウントで認証

### Q. デプロイ後にページが表示されない

- 本番サーバーでサーバーが起動しているか確認（タスクマネージャーで `pythonw.exe` を探す）
- ポート5000が他のアプリに使用されていないか確認
- Windowsファイアウォールでポート5000の受信が許可されているか確認

### Q. DBを元に戻したい

本番サーバーの `C:\App\ScheduleManagement\db\` にバックアップファイルがあります：

```cmd
cd C:\App\ScheduleManagement\db
copy web_app.db.bak_20260325_143012 web_app.db
```

> サーバーを停止してからリネーム → 再起動してください。
